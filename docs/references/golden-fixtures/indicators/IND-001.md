# IND-001 — Exponential Moving Average (period=3)

**Status:** active · v1 (2026-05-09)
**Category:** indicators
**Canonical:** `PythonDataService/app/engine/indicators/ema.py::ExponentialMovingAverage`

## What this pins

EMA values from `ExponentialMovingAverage` (period=3, k=0.5), validated against an independent pure-Python formula across 3 synthetic 8-bar price series (monotone, alternating, volatile).

## Oracle

hand_computed — SMA-seeded EMA written from first principles without calling `ExponentialMovingAverage`. Seeding and recursion implemented separately.

## Formula

```text
k = 2 / (1 + period) = 2 / (1 + 3) = 0.5

Warmup (samples 1..period):  EMA = mean(prices[0..i])  — running arithmetic mean
Post-warmup:                  EMA[n] = price[n] * k + EMA[n-1] * (1 - k)
```

### Critical seeding detail

The EMA seeds with the **running arithmetic mean** during warmup, not NaN. At samples=period, EMA = SMA(all period warmup prices). Post-warmup starts at sample period+1. This matches LEAN's `ExponentialMovingAverage.cs` (vendored at `references/lean/7986ed0aade3ae5de06121682409f05984e32ff7`).

## Reference

LEAN `Indicators/ExponentialMovingAverage.cs` (vendored commit `7986ed0`). SMA-seeded warmup; standard k = 2/(1+N) multiplier.

## Hand-Verification (Case A: prices=[10.0, 12.0, 14.0, 16.0, …])

| sample | EMA |
|--------|-----|
| 1 | 10.0 (mean of [p0]) |
| 2 | 11.0 (mean of [p0, p1]) |
| 3 | 12.0 (SMA seed = mean of [p0, p1, p2]) |
| 4 | 14.0 (16.0 × 0.5 + 12.0 × 0.5) |

All bars produce a float value. The EMA always returns a value during warmup (running SMA), so there are no `None` outputs.
