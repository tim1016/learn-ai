## One-paragraph narrative

We ported `SpyEmaCrossoverAlgorithm` from LEAN to our Python engine months ago and have a bit-exact backtest. To run it on IBKR paper, we needed a live runtime that shares the strategy class with the backtest — same code, same math, different driver. We built that runtime in this session: a `LiveEngine` that consumes IBKR's real-time bars, wraps the existing four-layer paper-safety boundary, and routes orders through the same `place_paper_order` path the broker UI uses. The proof is a replay-parity test that runs both engines side-by-side against the same fixture and asserts **exact** match (zero tolerance) on order count, fill prices, fill quantities, fees, equity curve, trade log, and insight scores. That test passes. The remaining work to get a real paper trade is operational, not engineering: confirm the IBKR market-data subscription is active, log into Gateway, run the runner.

## Q&A preparation

### "How do you know the live runtime won't drift over the paper week?"

Three layers of containment:
1. The replay parity test runs in CI on every PR — drift in the runtime would break it instantly.
2. The reconciliation report runs nightly during paper week. Any divergence outside cent-tolerance is classified (`precision`, `timestamp`, `fill_model`, `slippage`, `commission`, `data_gap`, or `unknown`) and the unknown classification fails the gate.
3. The strategy class is unchanged — same code that's bit-exact against LEAN today.

### "What if the broker drops the connection mid-day?"

The runner has a reconnect loop in the existing `IbkrClient`. We've documented the IBKR Gateway daily-restart in `docs/architecture/ibkr-integration-tdd.md` §6 risk register. Recommended: a manual mid-week reconnect test before relying on it for production.

### "What happens if the strategy fires an order but the next minute bar's open is way off the consolidated close?"

That's exactly the slippage the reconciliation report measures. It's the residual gap between "ideal next-bar-open fill" (what the backtest assumes with `FillMode.NEXT_BAR_OPEN`) and "actual broker fill on the next print after submission." Tolerance: `atol=Decimal("0.05")` for fill price, ±5 seconds for fill time.

### "Why not just use LEAN's live deploy?"

Two reasons:
1. **Repo philosophy.** We mine LEAN for math, port it, and drop the dependency. The math registry (`docs/math-sources-of-truth.md`) names `PythonDataService/app/engine/` as canonical, with LEAN listed as a vendored reference.
2. **Sovereignty.** If LEAN changes upstream, our system shouldn't. Our pinned port is the production artifact.

LEAN-via-IBKR is still useful as a shadow/comparator run for triangulation; we've left that door open. It's just not the receipt target.

### "What's the smallest change between backtest and live?"

`FillMode.NEXT_BAR_OPEN` is the parity target. In the backtest, market orders fill at the next 1-minute bar's open. In live, they fill on the next print after submission. The replay test pins the backtest exactly; the reconciliation step measures the residual gap.

### "Sizing — what fraction of the account is at risk?"

The strategy runs at native `set_holdings(SPY, 1.0)` — full allocation. We deliberately did NOT cap to e.g. 10% because changing the target fraction changes the integer share count via `int(target_value / price)`, which would break parity with the backtest in subtle ways. If a real risk cap is needed for the paper week, we generate a separate capped backtest baseline and reconcile against that — not the receipt run.

### "When can we run live (not paper)?"

That's Phase 4 in `docs/architecture/ibkr-integration-tdd.md` §7. The hard guard against live is intentional — even loosened later, it goes into a separate runner with its own config profile, not a flag on the paper runner. We do not put the highest-risk bypass next to normal deployment config.
