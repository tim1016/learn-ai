## Problem Statement

The bot control panel buries its only live, trustworthy surface — the Bot event stream — inside a workbench tab, while the page's prime real estate goes to a large lifecycle flow-chart and side panels that present reference material as if it were live actionable state. The operator pays for this daily:

- **Submit-time dead ends.** During the 2026-07-07 redeploy session, three bots in a row (`dep_val_smoke_002`, `DIagVal6`, `deiagAPPL6`) showed "Ready to deploy" in the Fresh run form, accepted a full action plan, and then failed at submit with "durably STOPPED. Resume the bot to clear the stop latch." The durable STOPPED latch was knowable before submit — the form just never asked.
- **False readiness.** On every bot in the session, the deploy form reported "Ready to deploy" (or a "Complete" Legs step) while the action plan's ON ENTER / ON EXIT legs were still empty. Nothing validates leg presence today, frontend or backend.
- **Invisible safety halts.** `Valgate-Jul6`, `DVS-SPY-SPY-0701`, and then `June23` accepted starts, died within seconds, and landed on Recovery lane · Poisoned with Prior Halt Trigger = Cold Start Divergence — yet Recent Incidents said "No recent incidents." A poisoned safety halt is invisible exactly where incidents are supposed to live, because boot-time reconciliation refusal never mints an OperatorIncident.
- **Deploy-time identity incoherence.** `DEPVALJUL1` (an MU bot) opened a Fresh run form prefilled with `signal_stream=SPY` and an enabled Deploy & start; `PrajiDemo` (AAPL context) got the same SPY prefill while the fleet table said flat and the detail page said ExposureUnknown. Nothing ties the instance's symbol, the signal stream, and the action-plan symbol together, and nothing blocks start on unknown or contradictory exposure.
- **The one good block is a dead end.** `Validation-Jul6` shows the right pattern — account NOT_PROVEN disabled Deploy & start *before* submit — but "Reconcile account" is prose, not a link, and the missing account-truth evidence is never named.
- **Scattered actionability.** Recovery guidance is split across an attention dropdown, a node inspector, disabled-button tooltips, and a guidance timeline. When a bot is blocked, the operator hunts across surfaces to find which one holds the action.
- **Clutter and lying controls.** The page does not fit the view; the flow-chart dominates; the same global guidance blocks repeat regardless of context. Four confirmed copy/control defects: the blocked-deploy alert renders "Deploy — blockeddeiagAPPL6…" with the space between the category and the bot-scoped detail missing; Resume rendered disabled as "Already running" on a detail page that simultaneously showed no live runtime; `June23` showed disabled-looking command titles while the actual button elements stayed enabled in the DOM; and the deploy form showed a successful start message beside a stale "already running" blocker.

## Solution

Reorganize the page around a strict division of labor — **nodes show values, the stream shows actions, the lower section explains**:

1. The **Bot event stream** (already built: spine events, gate-walk drill-in, terminal errors, SSE + backfill) is promoted out of the activity tab into the **side panel**, always visible next to the flow-chart. It is the live, actionable surface of the page.
2. Stream rows gain **inline actions**: when a row's condition has an operator action available right now (a poison/safety halt inside a run → recovery path, a run-time stop → Resume), the action renders on the row itself — live when the current verdict allows it, greyed with the backend's reason when it does not. The stream is run-scoped; blockers that precede any run belong to the deploy form's gates, not to rows.
3. **Node-scoped receipts render inside the flow-chart nodes.** The node is the inspector now: compact value at rest, full receipts on expand. The separate node-inspector pane is deleted (surface disposal: replace, don't add). The chart itself becomes visibly thinner and more compact.
4. The displaced non-node-scoped side-panel content (trader guidance proof lines, change-for-next-run, advanced evidence, audit trail) moves to a **lower documentation section** in normal page flow — plain scrolled reference material, no tabs, no overlays, no CTAs, no urgency styling.
5. The deploy form gains **pre-submit gates for every blocked path the audit hit**: the durable STOPPED latch, empty action-plan legs, symbol/signal-stream/action-plan identity coherence, and unknown or contradictory exposure — plus a deep-linked, evidence-naming account-proof block. A **safety-halt → incident bridge** makes poisoned halts (including Cold Start Divergence) first-class Recent Incidents with forensic detail.

This supersedes the layout portions of the Bot Control Dashboard PRD and the inspector-receipts PRD, and amends ADR-0024's surface hierarchy. The event-stream data contract (BotEventRow, GateStep, TerminalError, identity ladder, classifier seam) is unchanged and remains authoritative.

## User Stories

1. As a bot operator, I want the Bot event stream visible in the side panel at all times, so that the live narrative of what my bot is doing is never hidden behind a tab.
2. As a bot operator, I want the lifecycle flow-chart to be thinner and more compact, so that the chart, the stream, and the page controls all fit in one view without scrolling the page shell.
3. As a bot operator, I want a stream row whose condition is resolvable right now to carry the resolving action inline (e.g., a run's stop or safety-halt terminal row carries Resume or the recovery path), so that I act where I read instead of hunting for the right button elsewhere.
4. As a bot operator, I want an inline row action that is currently unavailable to render greyed with the backend's exact reason, so that I know why I cannot act and what would unblock it.
5. As a bot operator, I want a poison/safety-halt row to offer the recovery path (Fresh run) inline, so that post-run recovery has one obvious next step.
6. As a bot operator, I want destructive or irreversible actions launched from a stream row (Mark poisoned, Flatten and pause) to keep their typed confirmation flow, so that inline convenience never weakens the safety model.
7. As a bot operator, I want a resolved condition to appear in the stream as its resolving event, so that the history shows both the problem and the fix in one narrative.
8. As a bot operator, I want each lifecycle node to display its own receipt values inside the node, so that the node's shape and its numbers are one thing and I never cross-reference a separate pane.
9. As a bot operator, I want to expand a node in place to see its full receipts (headline, detail, source, evidence time), so that drill-down does not navigate me away or open an overlay.
10. As a bot operator, I want nodes with no emitted events to say "not emitted yet" honestly inside the node, so that empty never masquerades as healthy or borrows global data.
11. As a bot operator, I want the trader guidance, proof lines, change-for-next-run, and advanced evidence content to live in a lower documentation section in normal page flow, so that reference material reads as reference material and never competes with live surfaces for urgency.
12. As a bot operator, I want the documentation section to carry no CTAs and no live-state claims, so that when I see a button anywhere on this page I know it is real and current.
13. As a bot operator, I want the Fresh run form to detect the durable STOPPED latch the moment I check "Start trading immediately," so that I learn about the latch before filling in an action plan, not at submit.
14. As a bot operator, I want the form's primary CTA to switch to "Deploy only" (with copy explaining that Resume must clear the latch first) when the latch is present, so that the path I am offered is one that can actually succeed.
15. As a bot operator, I want the deploy readiness verdict to fail while required action-plan legs are missing or unsupported by the selected strategy/package, so that "Ready to deploy" is never a lie while valid entry-only or roll plans remain possible when the backend contract allows them.
16. As a bot operator, I want a Cold Start Divergence (or any poisoned safety halt) to appear in Recent Incidents with its forensic facts — halt trigger, source flag, evidence time, run id, and the artifact/log path — so that a bot that died for safety reasons is impossible to miss.
17. As a bot operator, I want a bot that dies after an accepted start to explain why on the page, with a pointer to the exact audit artifact, so that "it just went to Bot Off" never happens silently again.
18. As a bot operator, I want the safety-halt incident to be deduplicated by its incident key, so that one halt is one visible story, not a repeated alarm.
19. As a bot operator, I want the blocked-deploy alert copy fixed ("Deploy — blocked deiagAPPL6…", with the space), so that error text reads as authored prose, not string concatenation.
20. As a bot operator, I want raw backend identifiers (reason codes, gate ids, halt triggers) rendered through the shared receipt-label pipe everywhere on the redesigned page, so that I read trader language while opaque audit tokens stay exact.
21. As a bot operator, I want every timestamp on the redesigned surfaces rendered through the shared timestamp display component, so that time display is consistent and date-anchored values never drift a day.
22. As a bot operator, I want the top action toolbar (Resume / Pause / Stop / Flatten / Fresh run / Mark poisoned) to remain available for commands that have no triggering event (pausing a healthy bot), so that the stream's inline actions are shortcuts, not the only door.
23. As a keyboard or screen-reader user, I want inline row actions and expandable nodes to be reachable and labeled (WCAG AA, AXE-clean), so that the redesign does not trade accessibility for density.
24. As a trader reviewing a bot after the fact, I want the stream's backfill to show past terminal rows with their actions greyed (verdict-now says the moment has passed), so that history is preserved without advertising stale affordances.
25. As a developer or agent maintaining the panel, I want the row→action mapping to be a closed, exhaustiveness-tested set, so that a new terminal code cannot silently ship without a decision about its inline action.
26. As a developer or agent maintaining the panel, I want the node-inspector pane, the trader-guidance timeline, and other superseded surfaces deleted (not hidden), so that the codebase matches the surface-disposal rule: one verdict surface, one stream, everything else a projection or gone.
27. As a bot operator, I want the Fresh run form to refuse Deploy & start when the instance's symbol, the prefilled signal stream, and the action-plan symbol disagree (an MU bot prefilled with a SPY signal stream), unless I explicitly confirm the new run's identity with the inherited symbol's provenance shown, so that a lineage failure cannot silently deploy the wrong strategy.
28. As a bot operator, I want Fresh run start blocked when the fleet table and the detail page disagree on exposure, or exposure is unknown, so that I never start trading on top of an unproven position.
29. As a bot operator, I want the account NOT_PROVEN block to deep-link "Reconcile account" to the exact reconciliation view and name the missing account-truth evidence, so that the one pre-submit block that already fires correctly is also actionable instead of a dead end.
30. As a bot operator with no run bound (fresh instance, or every deploy attempt blocked pre-run), I want the side panel to say honestly that no run exists yet and route me to the Fresh run form, so that an empty stream is never mistaken for a healthy quiet one.
31. As a bot operator, I want the side-panel stream to read as a vertical feed with no horizontal scrolling at panel width, so that promoting the stream does not trade the wide tab's readability for scroll soup.
32. As a keyboard or screen-reader user, I want node expansion driven by a dedicated header button with the expanded receipts in a separate labelled region — one node open at a time, focus preserved — so that the in-node inspector never nests interactive controls inside a clickable card.
33. As a bot operator, I want an action's disabled reason to agree with the evidence on the same page — never "Already running" while the runtime section shows no live process — so that a disabled control teaches me the true state instead of lying about it.
34. As a bot operator starting a Fresh run on a bot whose previous run died in a poisoned safety halt, I want a pre-submit advisory naming the prior halt trigger and linking the reconciliation view, so that I can resolve the divergence before the new run predictably halts the same way.
35. As a keyboard or automation user, I want unavailable command buttons to be semantically disabled and non-dispatching, not merely styled or titled as disabled, so that the UI's affordance contract is true in the DOM as well as on screen.
36. As a bot operator whose Deploy & start was accepted, I want the form to transition to the accepted run's live/in-progress state instead of showing a success message beside an "already running" blocker, so that post-submit feedback is one coherent story.

## Implementation Decisions

- **Division of labor (locked):** nodes show values, the stream shows actions, the lower documentation section explains. No surface does another surface's job.
- **Stream promotion, not stream construction.** The existing Bot event stream component (SSE live tail + paginated backfill, row expansion to gate-walk and terminal-error forensics) moves from the activity tab into the persistent side panel. The BotEventRow / BotEventRaw / GateStep / TerminalError contracts are unchanged.
- **Inline actions are a frontend join to verdict-now.** A stream row carries only its condition identity (terminal-error code, gate id, event type). A small pure mapper — a closed affordance map from condition identity to operator command — is joined at render time against the operator surface's actions block (enabled / disabled_reason_code / gate_results) to decide whether the action renders live or greyed-with-reason. No BotEventRow contract change; the stream stays purely historical ("one source, two projections" per ADR-0024 — the stream never re-derives the verdict). Unmapped terminal codes render no action and fail the exhaustiveness snapshot test.
- **The affordance map's vocabulary is closed on both sides.** Domain: the closed set of spine event types and terminal-error codes. Codomain: the existing closed lifecycle action-id set (start-process, resume, pause, flatten-and-pause, stop, mark-poisoned, redeploy) plus exactly one navigation affordance — open the Fresh run form. "Deploy only" is a CTA state of the deploy form, not a lifecycle action id; the map introduces no new command vocabulary.
- **Inline actions invoke the same operator commands that exist today.** No new backend command paths. Typed confirmation flows are reused for destructive commands.
- **Run-scoped rows vs instance-scoped blockers.** The stream is run-scoped: the side panel binds to the live run, else the evidence run. Inline actions therefore exist only for conditions that produced a run event (a poison halt inside a run carries its recovery action). Blockers that precede any run — the durable STOPPED latch on a fresh deploy attempt, account NOT_PROVEN, deploy-form incoherence — never have a stream row; they are verdict-now concerns owned by the deploy form's pre-submit gates and the toolbar. When no run id resolves, the side panel renders an honest-empty "no run bound yet" state that routes to Fresh run — it never borrows another run's rows.
- **Top toolbar survives as the verdict-now command surface.** Locked decision "actionability lives in the stream" is interpreted as: one mechanism (operator-surface verdict), two placements (toolbar for operator-initiated commands with no triggering event; stream rows for condition-scoped shortcuts). Both render from the same actions block; neither invents availability.
- **In-node receipts.** Each lifecycle node renders its node-scoped receipts inside the node: at rest, a compact primary value plus status; on expand (click), the node grows in place to show full receipts (headline, detail, source, gate id, evidence time). The node-inspector pane is deleted. The inspector-receipts PRD's north star (node + receipt = two views of one backend receipt) is realized by collapsing the two views into one; its verified sourcing facts remain binding — halt trigger/at/detail come from the instance last-exit record (only when the poison flag is present), never from the prior-run classification; nodes without event emitters render honest-empty.
- **Chart rendering: extend the existing custom HTML/CSS chart.** Nodes are already HTML elements and can host receipt content directly. The chart is restyled thinner/more compact. The unused flow-chart library dependency currently declared in the frontend package manifest is removed. No new dependency.
- **Lower documentation section.** A plain page-flow section (not tabs, not overlays) houses the displaced reference content: trader guidance proof lines, change-for-next-run, advanced evidence, and the audit trail. Styled as documentation: no buttons, no CTAs, no severity coloring. Raw codes go through the receipt-label pipe; backend-authored prose is rendered verbatim. This does not demote real remediation affordances; those must move to a verdict-now surface before the old placement is deleted.
- **Surface disposal executed in the same change:** the node-inspector pane is deleted; the trader-guidance "recent activity" timeline is deleted (redundant with the promoted stream); the attention dropdown's explanatory prose moves to the documentation section. Its remediation CTA is retired only after each backend-authored remediation has a replacement in the stream, toolbar, deploy form, or account/reconcile action surface. In particular, `reconcile_instance` remains available as a verdict-now affordance until a replacement ships. The runtime banner (incident headline) is retained as a projection per ADR-0024 §9.
- **Deploy form pre-detection.** The Fresh run form reads the already-exposed verdict-now state when "Start trading immediately" is checked: durable desired-state evidence (`desired_state.state === STOPPED`) and the host-process start capability / gate rows (`STOPPED_REQUIRES_RESUME`) are the latch sources, not the Resume action's disabled reason, because Resume can be independently blocked by broker-safety proof. Deploy & start is disabled and the primary CTA switches to Deploy only, with copy explaining that Resume clears the latch first. The backend submit-time gate is unchanged and remains authoritative.
- **Action-plan readiness gate.** The deployment readiness verdict gains a backend check for the defect actually observed: a fully empty plan, missing required ON ENTER entry legs, or a plan shape the selected strategy/package cannot consume yields a blocked-before-submit verdict with a dedicated reason code. Missing ON EXIT close legs hard-block only when the selected package explicitly requires a close leg (for example the deployment-validation contract); otherwise the existing parity warning path for valid entry-only/roll plans stays non-blocking unless runtime consumption is unsupported. The frontend's blocked-reason computation mirrors the backend verdict; the frontend never claims readiness the backend would refuse.
- **Deploy-time coherence gates.** Three further pre-submit checks from the audit, same family as the latch and action-plan checks, each with its own reason code: (1) *identity coherence* — when the instance's symbol, the prefilled signal stream, and the action-plan symbol disagree, Deploy & start is disabled until the operator explicitly confirms the new run's identity, with the inherited symbol's provenance shown; (2) *exposure coherence* — when exposure is unknown or the fleet table and detail page disagree, Fresh run start is blocked pending explicit confirmation; (3) *prior-halt recurrence advisory* — when the prior run exited in a poisoned safety halt (Cold Start Divergence, twice in the session), a non-blocking advisory names the prior halt trigger and deep-links the reconciliation view: Fresh run stays the sanctioned recovery path, but the operator learns the divergence is unresolved before the new run predictably halts on it. The account NOT_PROVEN block — the one gate that already fires pre-submit — becomes actionable: "Reconcile account" deep-links to the reconciliation view and names the missing account-truth evidence (composes with the account-triage/reconciliation PRD's recovery lenses).
- **Side-panel stream layout contract.** The stream's current rendering is a wide table with a large minimum width — unusable at half-page width. In the side panel it renders as a vertical feed/list: stacked rows (headline and narrative at rest; gate-walk and forensics on expand), inline actions wrapping beneath the narrative, no table semantics, no horizontal scrolling at panel width, and internal vertical scroll bounded by the panel's height. The table layout is disposed of along with the activity-tab placement.
- **Node expansion accessibility contract.** Today's flow nodes are whole-card button-role elements, and receipt details render as interactive disclosure elements — nesting the latter inside the former is invalid. In the new chart: the node card is a non-interactive container; a dedicated node header button toggles expansion (aria-expanded, aria-controls); the expanded receipts region is a sibling region outside the button, labelled by the node; one node is expanded at a time; focus stays on the header button across toggle, and Escape collapses and restores focus. No interactive element nests inside another.
- **Safety-halt → incident bridge (exact contract).** Two incident systems exist today, and the bridge must span both: the Recent Incidents panel renders the run-scoped incidents endpoint, which parses the run's live log into incident records with a backend-classified category; the operator-incident store (ADR-0015) is a separate per-run file store that today feeds only the cockpit incident headline, filtered to the watchdog/order/submit categories. The bridge is three specific changes: (1) a poisoned safety halt — including Cold Start Divergence at boot-time reconciliation refusal, which today writes only the poison flag — mints an OperatorIncident through a new safety-halt incident builder, not the BotEvent terminal incident key path; the OperatorIncident category/notice code union gains `safety-halt`, and the safety-halt dedupe key is `(strategy_instance_id, run_id, halt_trigger, evidence_time_ms or last_exit_artifact_ref)` so boot-time poison flags need no evaluation/order identity, and the incident payload carries halt trigger, source flag, evidence time (int64 ms UTC), run id, and the artifact/log path; (2) the run-scoped incidents endpoint is extended to compose operator-incident store records with the parsed-log records into its existing incident-record contract, deduplicated, so the panel shows them with no frontend change beyond a copy-map entry for the new category; (3) the incident-headline category filter admits safety-halt.
- **Copy defect.** The blocked-operation alert's title/detail join is fixed so category and bot-scoped detail are separated by a space; error copy follows the repo's error-authoring guidance (required core + remediation only when the system truly knows the fix).
- **Disabled-reason consistency.** An action's disabled reason must agree with the runtime evidence in the same operator-surface response. The observed contradiction — Resume disabled as "Already running" while the page shows no live runtime — arises when the desired-state axis (already RUNNING) is narrated as if it were the process axis. The reason copy names the true axis and points to the applicable action (Start bot process), and never asserts a runtime state the same response disproves. This is a backend authoring defect, not a frontend patch.
- **Semantic disabled state.** Any unavailable command rendered from the operator-surface verdict must be unavailable in the DOM as well as in copy: native buttons use `disabled`; non-button affordances use `aria-disabled` plus dispatch suppression and focus behavior appropriate to the element. A disabled-looking title, tooltip, or CSS class without semantic disablement is a defect.
- **Post-submit state coherence.** Once `Deploy & start` returns an accepted run id, the Fresh run form stops evaluating itself as an idle pre-submit form. It either routes/focuses the accepted live run state or renders a start-in-progress state for that run id. It must not show the accepted-start success message beside a stale "already running" deployment blocker.
- **ADR-0024 is amended in place** (inline-actions decision; §9 disposal table updated for the new side-panel placement, the deleted inspector pane, and the deleted guidance timeline). No new parallel ADR. The layout portions of the Bot Control Dashboard PRD and the inspector-receipts PRD are marked superseded by this PRD.
- **Definition of done for implementing PRs:** the bot-lifecycle account-owner authority document is updated in the same PR, per the existing rule.
- All temporal values remain int64 ms UTC end-to-end; display only through the shared timestamp component.

## Testing Decisions

- A good test asserts external behavior — endpoint response contracts and rendered operator-visible output — never internal signals, private state, or implementation order.
- **Backend contract seam (primary emphasis).** Async HTTP endpoint tests against the FastAPI app (the repo's existing async-client + ASGI-transport pattern, as used by the operator-surface, bot-events, and incidents router tests):
  - Deployment readiness verdict returns blocked-before-submit with the new reason code when the plan is fully empty, lacks required entry legs, or lacks a required close leg for a package that explicitly requires one; it clears for valid entry-only/roll plans that carry only existing parity warnings (regression test written failing-first — this is the observed defect without over-blocking valid plans).
  - Operator surface exposes the durable-STOPPED latch pre-submit through desired-state/start-capability evidence even when Resume's own disabled reason is broker safety (regression pin on existing behavior the deploy form now depends on).
  - An operator-surface response with desired state RUNNING and no live runtime bound never emits a disabled reason asserting the bot is running; the reason names the desired-state axis and the applicable next action (failing-first — this is the observed "Already running" contradiction).
  - Boot-time poisoned safety halt (Cold Start Divergence path) mints exactly one deduplicated safety-halt OperatorIncident using the safety-halt key `(strategy_instance_id, run_id, halt_trigger, evidence_time_ms or artifact ref)`; the run-scoped incidents endpoint returns it composed with parsed-log records in the existing incident-record contract, exactly once; and the incident-headline filter admits it (failing-first — this is the observed "No recent incidents" defect).
  - Deployment readiness verdict blocks on identity incoherence (instance symbol vs signal stream vs action-plan symbol) and on unknown or disagreeing exposure, each with its dedicated reason code.
  - Bot-events rows carry the condition-identity fields (terminal code, gate id, event type) the frontend join requires — contract pin, no new fields expected.
- **Unit seam (new, small).** The row-condition → action affordance mapper: pure function, exhaustiveness snapshot over the closed terminal-code and event-type sets (mirrors ADR-0024's classifier exhaustiveness pattern); unmapped members fail the build, not the operator.
- **Frontend rendered-DOM seam (lighter).** Component specs on the existing DI-level fake-service harness for the bot-control page, asserting what the operator sees: stream renders in the side panel; a terminal row shows its action enabled or greyed-with-reason according to the faked verdict; the side panel with no bound run renders the honest-empty no-run state; a node renders its receipt values and expands in place per the accessibility contract (header button, sibling region, one open at a time); honest-empty node copy; deploy form disables Deploy & start and switches the CTA when the faked latch is present; an accepted Deploy & start response does not render success beside an already-running blocker; unavailable toolbar/row commands are semantically disabled and cannot dispatch; the documentation section contains no interactive controls. Prior art: the existing bot-control page and child-component specs with the fake live-runs service provided at DI.
- **Visual verification in the delivery gate.** Playwright-driven screenshots of the redesigned page at the target desktop viewport and the stacked <1280px mode, verifying the stream renders as a feed with no horizontal scrollbar, the thinner chart and side panel fit the page shell, and an expanded node stays within the chart pane. These are delivery-gate checks reviewed before PR, not pixel-diff CI assertions.
- Every defect family observed in the 2026-07-07 audit ships with a regression test that fails before the fix (durable STOPPED pre-detection, action-plan readiness without over-blocking valid entry-only/roll plans, missing safety-halt incident, account-proof deep link/missing-evidence copy, identity/exposure coherence, disabled-reason contradiction, semantic disabled-state mismatch, post-start success/blocker contradiction, copy join).

## Out of Scope

- Building an authoritative per-bot PnL source (Monitor-node behavior unchanged: trade-activity summary plus explicit "P&L not yet available").
- New backend command paths or changes to the gates themselves (gates are the safety model and untouchable; only duplicate visualizations are disposed).
- Out-of-band push / notification delivery for stream events (explicitly deferred by the bot-event-stream PRD pending a delivery-channel decision).
- Node-click cross-filtering of the stream (clicking a node expands the node; filtering the stream by gate id is a possible follow-up, not this PRD).
- Making the Bot event stream instance-scoped (composing rows across runs). The stream stays run-scoped; the honest-empty no-run state and the deploy form's pre-submit gates cover the pre-run altitude.
- Fleet list (/broker/bots) redesign beyond what PR #950 already shipped for fresh-run-only surfacing.
- Adopting a third-party flow-chart library (decided against; the unused declared one is removed).
- Backfilling incidents for runs that halted before the bridge ships.
- Live-money trading concerns.
