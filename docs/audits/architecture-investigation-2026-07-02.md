## 1. Executive summary — top 10 risks and opportunities

**Bottom line.** The architecture's *spine* is sound and, in several places, better than typical: the file-canonical + rebuildable-Postgres-projection substrate is exactly current best practice, the durable-submit protocol matches professional OMS "query-before-resubmit" discipline, the order-ownership model matches IBKR's own identity reality, and the operator-surface "judgment vs evidence" boundary (ADR 0013) is a genuinely strong idea that is mostly enforced. The *failures cluster at the edges* — recovery/panic paths, the transport security of the data plane, and the wiring between the account-truth a trader reads and the gate that actually blocks trading. None of the ADRs is wrong enough to supersede; the pressing work is (a) fixing a handful of safety-critical recovery/transport defects, (b) closing the observation-vs-enforcement gap in reconciliation, and (c) governance hygiene as the design has outrun its ADR text. Below, **R** = risk, **O** = opportunity.

1. **[R — Critical] The bot-control plane is an unauthenticated broker-control API on `0.0.0.0`.** Every mutation (deploy/start/stop/pause/flatten/mark-poisoned/order-mint) has no caller authentication, is LAN-reachable, and is triggerable by simple-request CSRF from any webpage the operator visits; the data plane also holds and forwards the *hardened* daemon's token, so reaching the data plane inherits the daemon's authority. Interim fix is small (bind `127.0.0.1` + one shared-secret header + `TrustedHostMiddleware`, reusing the in-repo daemon pattern — not an auth system). §6 C1/C2, §10 P0-1/P0-2.
2. **[R — Critical] The panic button is broken.** Emergency-flatten always exits 3 (never stamps `order_ref`, VCR-0020) and the audit file the cold-start reconciler reads is never written (§5.3 #11), so a post-emergency-flatten run *Poisons* on next boot. The one action a trader needs in a crisis does not work end-to-end. §10 P0-3.
3. **[R — Critical] Recovery-flatten can drive the account net-short.** `_recovery_flatten` sizes from a ~20 s-stale snapshot → duplicate SELL (observed live 2026-06-16, −1 SPY). §7 row 5, VCR-0019, §10 P0-4.
4. **[R — High] A frozen account cannot be cleared in production.** Restart-intensity and watchdog halts *write* the freeze, but `clear_account_freeze` has no shipped endpoint or CLI caller — recovery is currently file-editing only. Freeze is a one-way door. §3.3 #3, §10 P0-6.
5. **[R — High] The account-truth board a trader reads gates nothing.** `account_truth.final_verdict` is consumed by no submit or start gate; a *separate* fail-closed classifier enforces on the submit path. Two independent derivations that can disagree — a trader can see a clean board while submission is blocked, or vice-versa. This is the central reconciliation truthfulness risk (and the account-triage PRD's durable-receipt work is meant to close it). §3.3 #1, §10 P1-1.
6. **[R — High] A crashed sibling bot stays `ACTIVE` forever.** The instance registry has no liveness and no crash-retire path, so a dead bot's exposure is attributed to a live owner instead of surfacing as `retired_owner_live_exposure`. (Note: the *sibling-adoption* hole flagged in prior sessions is **refuted** — the classifier skips sibling namespaces before adoption; and the **S0 ephemeral-namespace** defect is **fixed** as of 2026-07-01. The liveness leak is what remains.) §3.3 #2, §10 P1-2.
7. **[R — Medium] The safety-critical intent WAL can self-poison after a crash**, because it lacks the torn-tail truncation its sibling broker-callback WAL already has; and the trader's *evidence trail* (parquet decisions/executions/trades + the run ledger) is the least crash-safe state in the system — non-atomic, no fsync, whole-file rewrite, and `executions.parquet` carries no `intent_id`. §5.3 #7/#8, §10 P0-5/P1-4.
8. **[R — Medium] ADR governance has outrun the text.** Five ADRs sit "Proposed" while carrying shipped-implementation amendments; a rename corrupted an enum literal inside an *Accepted* ADR (0015); and deferred ADRs are accumulating as unpaid debt while their subject matter ships anyway (adoption/reclamation, AccountOwner R3, action-plan engine consumption). Nothing is unsafe, but a reader cannot tell "decided" from "landed." §2, §12.
9. **[O] The architecture's spine is genuinely strong — and independently corroborated.** File-canonical + rebuildable Postgres projection matches Kleppmann/SQLite guidance and used ADR 0001's escape hatch *correctly*; the durable-submit protocol (persist-before-send, retry-cap-1, halt-on-uncertainty) matches FIX/Binance OMS practice; the ownership ladder (permId account-wide, `orderId` excluded, `orderRef` for-lifetime) matches IBKR's documented reality — and IBKR documents *no* orderRef length cap, which validates the platform's fail-closed 60-char self-test gate as exactly right. The durable-submit path is present-but-operator-gated-off pending one live cap-verification receipt, which is the correct posture. Keep and finish these. §5, §4, §11 (cap-C receipt).
10. **[O] The backend already authors the trader vocabulary, so truthful UX is mostly "render what the backend says."** `submit_readiness` codes, gate statuses, posture enums, notice tiers, and backend-authored receipts already exist; the UI work is to render them faithfully, burn down the acknowledged ADR-0013 client-side-judgment debt (`account-summary-attention`, execution-uncertainty codes), make destructive actions render their *actual* MutationAttempt terminal state (killing the optimistic-banner class of bug), and write a trader manual in the system's own language with a CI-checked glossary against the closed vocabularies. §8, §9.

**How to read the rest:** §2 dispositions all 17 ADRs; §3 maps every evidence source and which one authoritatively answers each trader question; §4 traces one order end-to-end; §5 critiques the file substrate and endorses the projection; §6 is the vulnerability table; §7 is the 32-row edge-case matrix; §8 is the UI/UX redesign; §9 the trader manual; §10 the P0–P3 roadmap; §11 the test plan; §12 the exact ADR amendments and new ADRs to write.

## 3. Architecture map — account/order/bot evidence sources

### 3.1 Evidence-source catalog (VERIFIED, reconciliation deep audit)

Grouped by scope. "ENF" = a submit/start gate reads it (fail-closed); "UI" = a projection/display reads it; durability noted.

**Run-scoped (per `run_id`, under `live_runs/<run_id>/`):**
- `run_ledger.json` — deploy identity (account_id, spec/QC provenance, live_config); `account_id` is hashed into `run_id`. Immutable. ENF+UI. *(writes non-atomically, no fsync — §5.2a)*
- `intent_events.jsonl` (IntentWal) — legacy direct-submit lifecycle: `PENDING_INTENT`(fsynced pre-placeOrder)→`SUBMITTED`/`ACK_FAILED_UNCERTAIN`→terminal; `ADOPTED_BROKER_ORDER`. Append-only, fsync, monotonic seq, corruption→Poison. ENF+UI. **The durable answer to "what did the last submit do?"**
- `broker_callbacks.jsonl` — raw IBKR execDetails/commissionReport, host-runner first-capture (ADR 0014), idempotency key `event_type|exec_id|perm_id|order_ref|status|exec_time_ms`. Append-only fsync. Audit.
- `reconciliation_receipt.json` — cold-start/runtime reconcile outcome; `in_progress` sentinel→pass/fail so a crash never leaves stale `passed`. TTL **24h** (`config.py:161`). ENF (Resume guards)+UI.
- `verdict_snapshot.json` — child-written broker safety verdict each check (`observed_at_ms_utc`). ENF (Resume)+UI. **The per-run authority for paper/live.**
- `engine_runtime.json` — 1 Hz process liveness/posture; >30s stale → `EXITED_UNMANAGED`, wrong boot_id → `ORPHANED_CONTROL_PLANE`. "A sidecar alone never proves a process is alive." ENF (freshness gates posture)+UI.
- `poisoned.flag`/`halt.flag`/`run_status.json` — durable terminal flags; poisoned = redeploy-only. ENF+UI.
- `session_metadata.json` — verified account pair + `connection_epoch`. Forensic.

**Instance-scoped (per `strategy_instance_id`):**
- `live_state/<sid>/live_state.json` — engine's namespace-attributed `expected_position_by_symbol` (the running tally, "never decomposed from the net account snapshot"). ENF+UI. **The authority for "what does this bot own?"**
- `live_state/<sid>/desired_state.json` — durable RUNNING/PAUSED/STOPPED, versioned. ENF+UI.
- `live_instances/<sid>/broker_activity.jsonl` — per-instance activity WAL, survives redeploys; observation/audit-only (ADR 0014). UI.

**Account-scoped (per account, under `accounts/<ACCT>/`):**
- `instance_registry.jsonl` — **durable** append-only registry of allowed instance/run/namespace/lifecycle bindings; DEPLOYED (daemon deploy) / ACTIVE (start) / RETIRED (spawn-failure only). fsync+lock, no TTL, **no liveness**. ENF (submit binding gate, reconcile namespaces, classifier)+UI. **The durable namespace source of truth.**
- `account_events.jsonl` — typed forward-write audit (AccountOwner submit events, freeze recorded/cleared, restart-intensity breach, owner-generation, receipt recorded); strict read for safety, tolerant read for projection. ENF+UI.
- `unresolved_exposure.flag` — account freeze; active iff `cleared_at_ms is None`; no TTL. Written by restart-intensity breach + watchdog unsafe-halt; **cleared only by `clear_account_freeze` which has no shipped caller** (see below). ENF (deploy/start/resume/submit; exposure-reducing orders exempt)+UI.
- `owner_generation.json` — AccountOwner fencing generation + phase (accepting/reconnecting/draining/frozen). ENF (submit)+UI. *(per-runner-process last-writer-wins — no cross-process CAS)*
- `account_reconciliation_receipt.json` — account-level receipt wrapping Account Truth; TTL **60s**. **UI/observational only — no submit or start gate reads it.**
- `fleet_baselines/<ACCT>.json` — operator-approved flat-account reset (requires positions==[] and open_orders==[]). ENF.

**Ephemeral (not persisted / process-memory):**
- **Account Truth response** — composed per request from broker snapshot joined to the *durable* registry (fixed 2026-07-01, `cabab748`); `generated_at_ms` stamped. UI + input to the account receipt.
- **IbkrClient in-memory state** (`connected_account`, ib_async caches) — child client is the enforcement authority; data-plane singleton is advisory.
- **Publisher registry** (in-memory singleton) — lost on restart; consumed by enforcement only as the reconnect-recovery submit halt.
- **Order idempotency cache** (per `client_order_id`) — process-lifetime only, does not survive restart.

### 3.2 Trader question → authoritative source (VERIFIED)

| Trader question | Authoritative source | Endpoint | Authoritative or re-derivation? |
|---|---|---|---|
| **Is this bot running?** | host-daemon live process table + `engine_runtime.json` freshness | `GET /live-instances/{sid}/status` | Authoritative composite (sidecar alone is *not* proof of life) |
| **Allowed to submit?** | the submit-time gates themselves (freeze→registry→verdict→order_ref in `LivePortfolio`/`AccountOwner`/`place_paper_order`) | status `submission_capability`/`actions` is a **projection re-checked server-side before actuation** | Gates authoritative; DTO is a re-derivation |

| **Positions owned by this bot?** | `live_state.json.expected_position_by_symbol` (namespace-attributed tally) | `/status` | Authoritative. Account-Truth position attribution is a **UI heuristic re-derivation** (con_id join) |
| **Orphan/manual/outside positions?** | Account Truth board (owner classes + blockers) for UI; startup unexpected-position halt + classifier Poison for ENF | `GET /broker/account-truth` | **Two derivations** — UI board vs enforcement classifier; can disagree |
| **Unknown open orders?** | same split — Account Truth `unknown_open_orders` (UI) vs cold-start/runtime classifier over `list_open_orders` (ENF) | `/broker/account-truth` | Two derivations |

| **Safe to start/restart?** | layered chain: data-plane `_assert_start_allowed` → daemon start (dup-lock+binding+intensity+freeze) → `cmd_start` (identity+unexpected-position+**mandatory cold-start reconciliation receipt**) | `POST /runs/{id}/start` | Authoritative; capability projection is stale-able, server re-checks |

**Key architectural observation.** For six of the nine questions the authoritative source is a *fail-closed gate* and the UI shows a *re-derived projection that the server re-checks before acting* — this is the ADR 0013 boundary working correctly. The two genuinely worrying seams are (a) **orphan/unknown-order** where the UI board and the enforcement classifier are *independent derivations that can disagree* (the account-truth verdict is observational; a *parallel* fail-closed sweep guards submit — §3.3), and (b) **position ownership** where the trustworthy answer (`expected_position_by_symbol`) is engine-authored but the *account-truth* board re-derives ownership by con_id join, a heuristic.

### 3.3 The account-truth enforcement gap (CONFIRMED, updated)

Prior sessions flagged an **S0**: the account-truth board read *ephemeral* publisher-registry namespaces while enforcement read the *durable* `instance_registry.jsonl`, so a stopped bot's own completed orders showed as `foreign_or_unclaimed`. **This is REFUTED as of master** — fixed 2026-07-01 in `cabab748`: the board and receipt now load bindings via `read_account_instance_registry`, and registry-unavailable degrades to a *critical evidence gap* → `not_proven` rather than silently empty (`account_truth.py:34-39,91-128,1243-1246`). The authority doc already reflects this (`:172`). **The relevant memory note is now stale and should be updated** (see §12 memory action).

What **remains** true and load-bearing:
1. **`account_truth.final_verdict` is consumed by no submit or start gate** (grep-verified: only `AccountReconciliationService`, whose receipt serves GET/triage only). The board is observational. A *parallel* fail-closed fact-check does guard the submit path — the AccountOwner classifier runs its own live open-orders+executions sweep and poisons/freezes on unknown state (`account_classifier.py:133-341`) — but it is a **separate derivation**, so the board a trader reads and the gate that blocks submission can disagree. **This is the reconciliation architecture's central truthfulness risk**, and it is exactly what the account-triage PRD's durable-receipt work is meant to close.
2. **Crashed sibling stays ACTIVE forever — CONFIRMED.** `RETIRED` is written only on daemon spawn-failure; `stop()` and crash/exit paths never rewrite the binding (`host_daemon.py:451-456` vs `:792-856`; `run.py` writes only ACTIVE). A dead sibling's namespace stays "recognized" (skipped, unsupervised) in every other run's reconcile, and Account Truth keeps attributing its exposure to an ACTIVE owner instead of raising `retired_owner_live_exposure`. **The trust leak is the registry's lack of liveness, not classifier adoption** (the adoption hole flagged in prior sessions is *refuted* — the classifier recognizes-and-skips sibling namespaces before any adoption logic, `reconciliation_classifier.py:127-135,175-179`).
3. **`clear_account_freeze` has no shipped endpoint or CLI caller** (grep-verified: only the function + tests). Restart-intensity and watchdog halts *write* freezes, but there is **no production path to clear one** except editing files — recovery is currently manual/test-only. This is an operability P1 (see §10).
4. **AccountOwner is per-runner-process, not account-wide.** The submit lock (`asyncio.Lock`), phase, and generation counter live in one runner process; two runners on one account each write `owner_generation.json` last-writer-wins with no cross-process compare-and-swap. Cross-process protection today comes from the registry gate (duplicate active namespace → block) + the per-instance daemon start lock — **not** from an account-wide single writer. The R3 daemon/IPC is not shipped (authority doc `:404`).

## 5. File-backed persistence critique and DB-projection recommendation

### 5.1 The verdict up front

**Keep the file substrate canonical; the ADR 0001 decision was correct and has aged well.** The independently-researched industry pattern is exactly what this repo does: an append-only, fsync-durable event log is the primitive, and query/dashboard views are *derived* from it (Kleppmann's "make the log the source of truth, derive everything else" [ddia-logs]; SQLite's own file-vs-DB criteria [sqlite-appfile]; SEC 17a-4's 2022 audit-trail alternative accepting append-only versioned history that can reconstruct any prior state [sec-17a4]). The repo already took the one graduation step the evidence supports — a **rebuildable Postgres projection**, not a substrate migration (lifecycle read model, EF migration `20260630023000`, `bot-lifecycle-account-owner-authority.md:174-251`). Files stay canonical; Postgres is truncatable; `/status` never depends on it. This is the right shape.

The critique below is therefore not "move to a database." It is: (a) record that an ADR 0001 trigger fired and govern the next projection slice; (b) close specific durability/mechanics gaps in the file layer itself; (c) name what the projection should and must not claim.

### 5.2 What the file layer does well (verified directly)

- **The intent WAL is textbook.** `PENDING_INTENT` is fsynced *and* the parent dir is fsynced before `placeOrder` can run; `seq` is consumed before fsync so a crash-retry uses `seq+1` rather than re-writing a duplicate-seq line that would poison the next read; reads tolerate exactly one torn trailing line (an un-fsynced tail proves `placeOrder` never ran) and raise `IntentWalCorruptError` on any other malformation (`intent_wal.py:117-129,151-178`, VERIFIED). This satisfies the fsync-before-ack + torn-line-tolerance + atomic-append norms from the research [pg-wal], [lwn-atomic], [jsonl-practices].
- **Atomic snapshot writes** use tmp+`os.replace` for the daemon token and pydantic artifacts (`daemon_auth.py:125-132`; `artifact_io.atomic_write_pydantic_artifact`), the standard crash-safe rename pattern [lwn-atomic].
- **Content-addressed run identity** (`run_id = sha256(canonical_json(...))`, ADR 0006) gives idempotent deploy for free — an identical re-POST is a no-op, a changed input is a new run.
- **Account events are a typed forward-write envelope** with monotonic per-account `seq` assigned under an append lock, strict `int64 ms UTC` validation, and *strict-read-for-safety / tolerant-read-for-projection* separation (`account_artifacts.py`; authority doc §"Account Event Hardening Snapshot"). Safety folds fail closed; projection tolerates legacy rows without fabricating timestamps.

### 5.2a Per-artifact durability matrix (VERIFIED, control-plane deep audit)

The durability discipline is **inconsistent across artifacts** — the submit-critical spine is hardened, but two important artifacts write non-atomically with no fsync.

| Artifact | Write pattern | fsync | Atomic | Corrupt-read behavior |
|---|---|---|---|---|
| intent WAL `intent_events.jsonl` | append; seq consumed pre-fsync | flush+fsync+dir | append (no rename) | 1 torn trailing line tolerated; else `IntentWalCorruptError`→Poison |
| broker callbacks WAL | append; **truncates torn tail before appending** | yes | append | `BrokerCallbackWalCorruptError` |
| live-state sidecar | full rewrite | yes | tmp+fsync+`os.replace`+dir under lock | raise→Poison |
| desired_state.json | RMW, `version++` | yes | tmp+fsync+replace+dir under lock | `DesiredStateCorruptError`→start refuses |
| command files | one file per seq | yes | tmp+fsync+`os.replace` under dir lock | `CommandChannelCorruptError`→engine halts, file kept |
| account artifacts (events/freeze/registry/owner-gen) | locked append or tmp+replace | yes | yes, under `_file_lock` | strict reader raises; tolerant reader (projection only) skips |
| reconciliation_receipt.json | `in_progress` sentinel→terminal | yes | tmp+fsync+replace+dir under lock | write-fail→startup refused (exit 3) |
| engine_runtime / daemon_lease / mutation_attempts | — | yes | `atomic_write_pydantic_artifact` | reader→None on malformed/forward-incompatible schema |
| **run_ledger.json** | **plain `write_text(canonical_json)`** | **NO** | **NO tmp, NO rename** | `read_ledger` raises on bad JSON |
| **decisions/executions/trades.parquet** | **read+concat+rewrite-whole-file** | **NO** | **NO** | pandas raises; **crash mid-rewrite can corrupt previously-durable rows** |
| readiness.json | `write_text`+`os.replace` | **NO** | rename only | reader→None silently (loss-tolerant by design) |
| poisoned.flag | `open("x")` exclusive-create, first-wins | no | `O_EXCL` | malformed **raises**, never ignored |

**No rotation/retention anywhere** — per-run WALs and `account_events.jsonl` grow unbounded (only the sidecar sizing audit is ring-capped at 200); the account-event append does an O(n) rescan per write.

**Replay determinism is sound**: folds are pure and idempotent, cursor is `seq` never wall-clock, `run_id` is a canonical-JSON sha256 that excludes `created_at_ms`, broker-activity rows rebuild from the callback WAL.

### 5.3 Gaps in the file layer (VERIFIED, control-plane deep audit + prior audits)

1. **No tamper-evidence (M2 in §6).** `desired_state.json`, the command channel, and freeze flags are plain files; existing hashing is for dedup/change-detection, not integrity. A same-host writer can move a bot. Proportionate fix is filesystem perms now; HMAC-with-daemon-secret or a prev-hash chain later if the threat model warrants (the research points at simple prev-hash chaining / CT-style logs for tamper-evidence [ct-rfc6962]).
2. **`account_events.jsonl` was historically untyped** (raw dicts, no required `ts_ms`/`seq` — bot-lifecycle-observability audit v2 :225); the 2026-06-30 hardening fixed *new* writes but by design does not backfill, so legacy rows carry `ts_ms_resolved=false`. Acceptable, but the read side must keep treating unresolved timestamps as unorderable (it does).
3. **`executions.parquet` timestamp is wall-clock, not `exec_time_ms`** (VCR-P3-L, deferred) — a schema fidelity gap for any cross-engine reconciliation that keys on execution time.
4. **orderRef cap enforced only at submit, after ledger mint** — `build_order_ref` fails closed over cap 60 (`order_identity.py:136-137`, VERIFIED) but the deployment review hit `OrderRefTooLongError 71/72 > 60` only when the broker call was attempted, after the run was already minted (review :64). The `max_strategy_instance_id_len` helper (=25) exists but isn't enforced at deploy/mint. Fix: validate the instance-id budget at deploy time.
5. **orderRef cap C is still unverified against a live echo** — `DEFAULT_ORDER_REF_MAX_LENGTH = 60` carries a `TODO(#446 Acceptance Gate #1)` and durable-submit *activation stays refused* until a real paper order proves IBKR echoes the full string untruncated (`order_identity.py:40-49`, VERIFIED). This is the correct fail-closed posture, but it means the entire ADR 0008 durable-submit path is still operator-gated off in production (consistent with VCR-0002 "structural-complete-operator-gated").
6. **Multi-writer boundary is per-process, not per-account.** The submit lock is a process-local `asyncio.Lock`/`_submit_lock` inside one runner (`live_engine.py`; authority doc §2 "does not prevent a sibling or stale runner from calling IBKR"). AccountOwner V1 makes the *lane* single-writer per instance but the R3 single-writer-per-account daemon is not shipped. This is the real concurrency exposure — see §4 and the AccountOwner ADR recommendation (§12).

7. **`run_ledger.json` and the three parquet artifacts write non-atomically with no fsync** (matrix above, VERIFIED `run_ledger.py:274-284`, `artifacts.py:317-374`). The ledger is write-once so the window is tiny, but the parquet path is read-existing + concat + **rewrite-whole-file in place** on every flush — a crash mid-rewrite can corrupt rows that were already durable. The only mitigation is cursor ordering (flush parquet before advancing the sidecar cursor, `live_engine.py:1580-1621`). These are the trade/decision/execution history — the trader's evidence trail — and they are the least crash-safe artifacts in the system. **Fix: route both through `atomic_write` (tmp+fsync+rename); for parquet, write a new segment file per flush rather than rewriting the whole file** (segmented append is also the pandas-scale-friendly pattern).

**New durability bugs surfaced by this audit (not in prior audits, VERIFIED code paths):**

8. **IntentWal lacks the torn-tail truncation that BrokerCallbackWal has.** After a crash leaves a torn trailing line, `read_tail` *tolerates* it — but the first new append writes `json+"\n"` after the torn bytes, merging into one newline-terminated garbage line, so every subsequent read raises `IntentWalCorruptError` → next boot Poisons the run. Compare `broker_callbacks.py:116-129` (`_truncate_tolerated_tail`) against `intent_wal.py:117-129` (no equivalent). **Fix: port the truncate-tolerated-tail step into IntentWal.append.** This is a latent self-inflicted-poison on the safety-critical WAL.
9. **Dual IntentWal instance seq-cache collision.** Runtime RECONCILE constructs its own `IntentWal` on the same file (`reconciliation_orchestrator.py:157`) while the portfolio holds another with a warm `_next_seq` cache (`intent_wal.py:180-184`). A mid-run adoption append advances the file past the portfolio's cache; the portfolio's next append writes a duplicate/lower seq → non-monotonic → corrupt-on-read. `_inhibit_submits` prevents concurrent writes but not the stale cache afterward. **Fix: invalidate the portfolio's `_next_seq` after any external append, or use a single shared IntentWal instance.**
10. **ADR-0008's sidecar WAL cursors are inert.** The only steady-state envelope writer omits `last_intent_wal_seq` / `last_broker_callbacks_wal_seq` / `last_broker_activity_wal_seq`, resetting them to 0 on every flush (`run.py:944-960`). Correctness survives because fold-over-empty-projection replays the whole per-run WAL idempotently, but the designed incremental-cursor optimization is dead code. Either wire the cursors or amend ADR 0008 to record that full-replay is the shipped strategy.
11. **`emergency_flatten_audit.jsonl` has a reader but no writer.** The orchestrator consumes it (`reconciliation_orchestrator.py:194-204`) but `cmd_emergency_flatten` writes only free-text `emergency_flatten.log` (`run.py:2400-2407`). Consequence: post-emergency-flatten executions under `learn-ai/eflat-{account}/v1` are neither owned nor sibling namespaces at next cold start → `unknown_namespace` Poison until an operator stamps a fleet-reset baseline. Fail-closed, but the designed audit channel is unbuilt — this compounds VCR-0020 (emergency-flatten historically always failed the order_ref gate). **Fix: write the structured audit the orchestrator already reads.**
12. **`executions.parquet` has no `intent_id` column and synthesizes `perm_id=int(order_id)`, `exec_id=f"engine-{order_id}"`** when a fill event lacks broker ids (`live_engine.py:1472-1516`; `artifacts.py:81-104`). The fill↔intent join lives only in WAL/broker-activity land; the parquet trail cannot be joined back to intents on its own. Combined with `client_order_id` restarting from 0 every process (`live_portfolio.py:362,511`), cross-run identity in the parquet trail is reconstructable only via perm_id.
13. **Two crash candidates on recovery paths** (INFERRED bug, mechanics VERIFIED): `zip(pending_snapshot, acks, strict=True)` at `live_engine.py:1732` — the `RECOVER_ADOPT` outcome breaks without appending an ack, so a successful adoption yields fewer acks than pending → `ValueError` crashes the bar loop right after recovery; and `int(result.order_id)` at `live_portfolio.py:932` raises `TypeError` if an "accepted" AccountOwner result carries `order_id=None`. Both are on the least-tested recovery surface (matches the prior-audit theme: recovery/panic paths are the most-broken).

### 5.4 DB-projection recommendation

**What stays file-backed (canonical, never projected as source of truth):** run ledger, intent WAL, broker callbacks WAL, desired state, command channel, reconciliation receipts, account artifacts (instance registry, freeze flag, owner generation, account events), poison/halt sentinels. These are the audit trail and the safety-critical state; they must remain fsync-durable files per ADR 0001 and ADR 0008's reaffirmation (0008:74).

**What is correctly projected to Postgres (rebuildable read model, already shipped):** cross-run/cross-account *operator timelines* and *safety-triage queries* — the `bot_lifecycle_events`, `account_lifecycle_events`, `operator_gate_snapshots`, `lifecycle_node_receipts`, `account_owner_status_snapshots` tables. This fired **ADR 0001 projection trigger #2** ("cross-run analytic queries become a hot path in the operator's daily workflow"): a trader asking "show me every warning/critical lifecycle row across my fleet" is exactly a cross-entity query the filesystem answers poorly. The shipped guardrails are right and must be kept: `LIFECYCLE_PROJECTION_ENABLED=false`/empty `POSTGRES_URL`/stale DB → fall back to the file path; "Postgres unavailable ≠ bot state unknown"; the projection may *not* claim R3 single-writer authority or "safe to submit" (`bot-lifecycle-account-owner-authority.md:227-251`).

**The one governance action owed:** amend ADR 0001 to record that trigger #2 fired, what shipped, and the rule that *every future projection slice names its trigger and its fallback in the ADR's ledger* (see §12 ADR 0001 amendment). Without that, the projection surface grows without the discipline that made the first slice safe. **Do not** project the submit-critical path (intent WAL, order identity, freeze state) into the DB as anything but a read mirror — the moment a UI infers "safe to submit" from a table, ADR 0008's fail-closed uncertain-ack semantics are bypassed. The authority doc already forbids this (`:244-246`); keep it forbidden.

## 6. Vulnerability findings

**Scope reminder.** No-auth-by-design is a deliberate current-state choice; this section proposes *interim mitigations that are not an auth system* (bind changes, one static shared secret reusing the existing daemon pattern, Host/Origin validation, file permissions). It does not design authentication.

**Headline.** The *host daemon* (port 8765) is well-hardened — mandatory 256-bit token, `hmac.compare_digest`, `0o600`, atomic write, list-form subprocess, three-layer path confinement. The *data plane* it fronts (FastAPI, port 8000) has **no caller authentication, binds `0.0.0.0`, and is reachable by simple-request CSRF from any webpage the operator visits** — and because the data plane holds the daemon token via the artifacts bind mount and forwards it, whoever reaches the data plane inherits the daemon's full deploy/start/stop authority without ever seeing the token. The hardened daemon sits behind an open door.

- **Subprocess construction is injection-safe** — every spawn is list-form, `shell=True` appears nowhere (VERIFIED grep), request-derived args are pattern-locked (`strategy` `^[a-z][a-z0-9_]{0,63}$`), child env is `os.environ.copy()` + PYTHONPATH/IBKR_* only, **no broker credentials injected per-request** (`host_daemon.py:973-989`).

## 8. UI/UX redesign — page-by-page IA and copy

### 8.0 Design principles (from serious broker/OMS/algo-ops tools + aviation human factors)

Each principle carries the citation that grounds it; all access-dated 2026-07-02.

1. **One master status is a lie the system should not tell — but exception-first routing is not.** ADR 0013's "no synthetic master status" is correct and matches SRE alerting doctrine (every page must be *actionable*; informational ≠ interrupt [sre-book]) and aviation's three-tier Warning/Caution/Advisory hierarchy where red is reserved and may not be spent on decoration [ecfr-25.1322]. Resolution: keep the four independent facts (PROCESS/INTENT/READINESS/BROKER) *and* add a backend-authored **attention lane** that answers only "does this need me now?" — which the backend already computes (`operator_surface.trader_guidance`, attention groups). The UI routes attention; it does not invent a status.
2. **Reserve red for confirmed-dead / act-now; give in-flight uncertainty its own hue.** IBKR TWS does exactly this — Light Blue = transmitted-not-confirmed, Pink = cancel-requested-not-confirmed are distinct from Red = confirmed-canceled [ibkr-colors]. The platform's `submit_outcome_uncertain` / `broker_state_unproven` states must render in an *uncertainty* treatment, never green (false safety) and never red (false death).
3. **Working vs filled are quantities, not a status word.** EMSX carries `EMSX_WORKING`/`EMSX_FILLED` as separate fields [emsx-api]. Orders views must show filled/remaining as numbers, not collapse a partial fill into a single "working" chip.
4. **One kill action = block-new + cancel-working, atomically, scoped, with a machine-verifiable receipt.** CME Globex Kill Switch and MiFID II RTS 6 Art 12 both define the operator capability as *immediate* cancel-all + block-entry, scoped by algo/owner, emitting rejection receipts [cme-killswitch], [rts6]. The platform's Flatten-and-pause is the right primitive; it must produce a receipt (it does — MutationAttempt) and the UI must show that receipt, not an optimistic banner.
5. **Provide the intermediate verbs.** freqtrade's `stopentry` (halt new entries, let exits manage) and FIA's documented "reduce-only" mode sit between "run" and "kill" [freqtrade-api], [fia-guide]. The platform's Pause (durable intent, stops new exposure, ADR 0010) is exactly this — surface it as a first-class, low-friction verb distinct from the high-friction Flatten/Stop.
6. **Every act-now alert carries its procedure; everything else is a receipt, not an interrupt.** Airbus ECAM pairs each alert with its on-screen procedure [ecam]; ADR 0015's `OperatorNotice` already mandates that every critical notice carries an `action`. Enforce it in render: a critical with no actionable next step is a contract violation, not a UI state.
7. **Friction proportional to severity; consequence preview names the target; verb-labeled buttons; type-to-confirm only at the top tier.** GitHub Primer / IBM Carbon / Shopify Polaris / Atlassian all converge here [primer-confirm], [carbon-remove], [polaris-actions], [atlassian-modal]. Map: Pause = one click + inline undo; Flatten-and-pause = consequence-preview modal (positions to close, est. notional) + verb button; Stop/Mark-poisoned = staged confirm; Mark-poisoned (irreversible run condemnation) = type-to-confirm.
8. **Decision evidence = rationale in trader language + inputs + timestamped receipt links.** ICO/Turing "rationale explanation" (reasons + data, non-technical) [ico-explain]; PagerDuty incident timelines where every row links its receipt [pagerduty-timeline]. This is precisely the receipts model ADR 0013/0016 already author — the UI's job is to render it filterable by actor/action/target/time, never to compute it.
9. **Paper/observe/live is environmental, worded, persistent, and machine-readable — never inferable.** TWS turns the paper login red + "Simulated" banner; TradingView carries mode in the broker connection name; QuantConnect exposes `BrokerageEnvironment` as a first-class enum [ibkr-paper], [tv-paper], [qc-brokerages]. The platform's `effective_posture` is the machine-readable property; the UI must show PAPER_EXECUTION / PAPER_OBSERVATION / UNSAFE persistently, and resolve the historically-dangerous "Paper radio = observe-only" wording (attention PRD :36-37). (Caveat: TWS uses red for paper, which collides with principle 2's reserved-red — prefer a distinct persistent non-red treatment for observe/paper so red stays for act-now.)

### 8.1 Copy principles (trader-friendly without lying about backend state)

- **Verbs for actions, states for facts.** Buttons say what they do ("Flatten and pause", "Stop bot", "Mark run poisoned"); chips say what is true ("Paper — orders enabled", "Broker state not proven"). Never a button labeled "OK/Yes".
- **Every disabled control names its reason, backend-authored.** The reason comes from `actions.<verb>.disabled_reasons` / `ActionCapability.disabled_reason_code`, rendered through the receipt-label pipe — never a client guess (ADR 0013, 0017).
- **Every warning names its missing evidence.** "Not proven" ≠ "broken": say *which* evidence is stale/absent and how to supply it ("Broker account not confirmed for this session — reconnect the expected account"). Sourced from `submit_readiness.code` + gate `operator_next_step`.
- **Raw codes are receipts, never primary copy.** `reason_code`, `gate_id`, `NO_LIVE_BINDING`, `RECONCILIATION_NOT_AVAILABLE` render as "Receipt: …" under the trader sentence, through `receiptLabel` (hard rule + ADR 0013/0017). Opaque audit tokens (intent ids, order refs, hashes, paths) stay verbatim.
- **Absence-of-proof blocks, and overrides are audited and expire.** Copy for freezes and overrides states the evidence gap, the recovery path (recovery proof *or* audited override), and that overrides expire (`account_artifacts.clear_account_freeze`, §6.2).
- **Mode is always on screen.** A persistent posture strip (Execution + Exposure chips) that cannot be dismissed; no view ever lets the trader forget whether orders can reach the broker.

Copy examples (trader sentence → receipt): "This bot is watching the market but will not place orders. (Receipt: submission_capability = READ_ONLY)" · "Can't resume — the broker account for this session isn't the one this bot owns. Reconnect account DU1234567. (Receipt: BROKER_SAFETY_UNKNOWN)" · "Last order's outcome is unconfirmed. The bot halted itself and will not submit again until you reconcile. (Receipt: submit_readiness = submit_outcome_uncertain)" · "Account frozen: 3 restarts in 5 minutes. Clear with a recovery proof or an audited override. (Receipt: account.restart_intensity = freeze)".

### 8.2 Destructive-action representation (the safety-critical surface)

| Action | Backend primitive | Friction tier | Consequence preview | Receipt shown after |
|---|---|---|---|---|
| **Pause** | durable `desired_state=PAUSED` (ADR 0010) | one click + inline undo | "Stops new orders; existing positions unchanged." | desired-state write confirmation |
| **Resume** | guarded write (4 gates: paper-only, capability, reconciliation, uncertain-intent — ADR 0011) | one click; **disabled with reason** when any gate fails | — (blocked reasons shown inline) | desired-state write + which gates passed |
| **Flatten and pause** | `PAUSED` then one-shot `FLATTEN_NOW` (ADR 0010) | consequence-preview modal + verb button | positions to close, est. notional, open orders that will be cancelled first | `MutationAttempt` state (CONFIRMED / OUTCOME_UNKNOWN / …) — never an optimistic "done" |
| **Stop** | instance retirement, does *not* flatten (ADR 0010) | staged confirm ("Stop does not close positions — flatten first if needed") | current exposure that will be left open | MutationAttempt + retirement receipt |
| **Mark poisoned** | one-shot `MARK_POISONED`, irreversible run condemnation | **type-to-confirm** (the run id or bot name) | "This permanently condemns the run; it cannot be resumed, only redeployed." | poison flag receipt |
| **Clear freeze** | recovery proof XOR audited override (§6.2) | audited-override modal requiring reason + approver | the freeze evidence being cleared; that the override expires | `account_recovery_proof_recorded` / `account_audited_override_recorded` |
| **Emergency flatten** | out-of-process CLI + `confirm:true` (separate from flatten-and-pause) | type-to-confirm + account echo | — | audit row (note VCR-0020: this path historically always failed — verify fixed before surfacing) |

**Anti-optimism rule (from prior audits).** VCR-0008 (RECONCILE no-op showing green), VCR-0021 (STOP banner that never clears), VCR-0018-B (stop `accepted=true` after 2s while the process lived) all share one root cause: the UI declared success the backend hadn't confirmed. The MutationAttempt state machine (ADR 0010 B1: PREPARED→DISPATCHING→{RESPONSE_CONFIRMED|OUTCOME_UNKNOWN}→{EFFECT_CONFIRMED|EFFECT_NOT_OBSERVED|NOT_PROVABLE|EVIDENCE_CONFLICT}) is the fix — the UI must render the *actual* terminal state, and `OUTCOME_UNKNOWN` must look uncertain (principle 2), not done.

### 8.3 Page-by-page information architecture

**Design goal: more comprehensive, not noisier.** The current surfaces overlap (Bot Control, Account Monitor, Orders, Reconciliation, Account Truth board all show account/order facts in different shapes) and, in places, the frontend still derives operational meaning the backend should author (the account-triage PRD §11.9 explicitly names `lib/account-summary-attention.ts` and the execution-uncertainty codes as live ADR-0013 violations to fold back). The redesign principle: **one fact has one home; every other view links to it; trader copy and engineer drill-down are separated by a disclosure, not by a page.**

**Cross-page truthfulness rule.** Because §3.3 shows the account-truth *board* and the submit *gate* are independent derivations, the IA must never let a trader read "clean" on one page while another blocks trading. Fix: a single backend-authored account-safety verdict (fail-closed) is the *only* source any page renders; the board becomes a drill-down of that verdict's evidence, not a parallel answer.

**Per-page IA (proposed):**

| Page / route | Purpose (the one question it owns) | Renders (backend-authored) | Poll / freshness | Overlap to remove |
|---|---|---|---|---|
| **Bot Overview** (`/broker/bots/:id`) | "What is this bot doing and is it safe?" | 4 fact-pills (PROCESS/INTENT/READINESS/BROKER) + `submit_readiness` chip + Act-now bar; posture strip (Execution+Exposure) always visible | ~4 s poll + SSE; every pill carries `as_of_ms` | Stop being the place account-wide facts live — link to Account Monitor |
| **Lifecycle / Workbench** (same route, chart) | "How did it get to this state?" | backend `lifecycle_chart` (9 canonical nodes) + node-scoped receipts inspector + `operator_actionability` banner | node evidence `ts_ms` shown | Node selection explains, never gates (ADR 0017) |
| **Activity** | "What did the broker actually do?" | broker rows + backend narrative + verdict chip (`expected`/`…caveat`/`unexpected`/`engine_only_pending`); folds/clusters | SSE for Activity | Don't recompute fills client-side; render `visible_row_id` merge |
| **Orders** | "What's working / filled / cancelled right now?" | working vs filled as *quantities* (not one chip); cancel-lifecycle rows | poll | Compact columns (Time/Symbol/Position/Side/Order/Avg fill/Filled) per attention PRD |
| **Account Monitor** | "Is the account safe, and who owns what?" | the single account-safety verdict + freeze banner + connected-account proof + per-namespace ownership; P&L shown as "not yet available" (honest, inspector PRD ID8) | account receipt TTL 60 s shown | Absorbs the Account-Truth board as a drill-down; owns the "clean/not-proven" answer |
| **Reconciliation** (→ rename "Recovery") | "What's wrong and how do I fix it?" | triage gate rows + affected bots + the closed `available_recovery_actions` set + evidence refs | on-demand + latest receipt | The one place destructive recovery actions live (clear-freeze, adopt, reconcile) |
| **Audit / Receipts** | "Prove why the system did X" | receipts, decision logs, incident timeline with per-row evidence links (PagerDuty pattern) | historical | Engineer drill-down below the fold on every page links here |
| **Configuration / Deploy** | "What is this bot, and deploy a new one" | immutable package (spec+hash, QC provenance split verified-vs-recorded), sizing preset, account **derived** not typed | — | Config is intent (ADR 0016); Audit is evidence — keep separate |

**Client-side judgment to remove (ADR-0013 debt — burn down, don't amend around):** `lib/account-summary-attention.ts` and the execution-uncertainty codes (named by triage PRD §11.9) must move to backend-authored projection fields; the readiness-transition classifier (`lib/classify-readiness-transition.ts`) and clock arithmetic (`lib/clock-sync.ts`) are the *allowed* closed-enum/clock derivations per ADR 0013 §4 and may stay. Any `disabled-reason-copy.ts` map must be a closed map synced to the backend snapshot, failing to a receipt-label on unknown codes — never a client-invented reason.

**Frontend audit findings (VERIFIED at `771d9573`) — the concrete work behind the IA above:**

*The receipt-label pipe is a generic prettifier, not a synced registry.* `receipt-label.pipe.ts` title-cases code-like segments and infers opaqueness from the label *suffix* (`id|hash|path|ref|url`); it has **no backend label map**, and it mangles PascalCase IBKR enums (`PreSubmitted`→"Presubmitted"), which is why surfaces leave IBKR `OrderStatus` unpiped. Opportunity: back it with the closed operator-reason-code map so unknown codes fail visibly rather than getting silently title-cased.

*Client-side judgment to remove (ADR-0013 violations, severity-ordered).* Confirmed the two the triage PRD §11.9 already named — `lib/account-summary-attention.ts` (adds a 4th `notice != null` disjunct beyond the ADR formula) and `account-truth-execution-history.component.ts:126-134` (client mints uncertainty codes from field null-ness into an "N uncertain" badge). **New findings not previously flagged:** `latest-signal-strip.component.ts:39-46` derives a colored tone by string-matching *open evidence content* (`ENTER|BUY|LONG`→ok) — the sharpest violation, because the vocabulary is not a closed contract enum (ADR §4 forbids deriving over evidence content); `broker-orders.component.ts:432-446` client-encodes cancel *eligibility policy* (there is no backend `ActionCapability` for ledger rows) and `:539/551/561` hardcode `tone:'warn'` ignoring backend `severity`; `overview-tab.component.ts:117-122` **repaints backend-authored `edge.status`** when it deems a node "blocking"; `bot-control-page.component.html:124` hardcodes a notice headline while the wire `OperatorNotice.title` is ignored; and the sidebar `broker-banner.component.ts:140-215` derives paper/live client-side while `health.safety_verdict.final_verdict` exists on the wire — so **two authorities render "paper" simultaneously** (sidebar vs the bot-control Broker-proof pill). Legitimate (keep): `classify-readiness-transition.ts`, `clock-sync.ts` (closed-enum/clock per ADR §4), and the backend-prose `lifecycle-action-button` (the amendment done right).

*Inverse failure — backend verdicts authored but never rendered.* `RuntimeBannerComponent` (the designated renderer for `operator_surface.runtime_freshness` and `incident_headline`) is **imported nowhere**; `last_exit`, and lifecycle `receipt_payload`/`evidence_refs` have zero readers. The backend already computes staleness/incident judgments the redesigned page silently drops — so the page has *no data-age chip* despite the backend authoring one. This is the cheapest truthfulness win: wire the existing component.

*The contract-sync guardrail is broken (this is edge-matrix row 28, confirmed and worse).* The operator-reason-code snapshot regenerator (`regenerate_operator_reason_codes_snapshot.py:62-72`) and its pytest still target the deleted `cockpit-v2/lib/` path, so the byte-equality guard **silently skips**; the two committed snapshots already differ; and CI never runs `tests/services`/`tests/operator` — so a backend code addition ships without the matching frontend copy and CI stays green. The contract spec itself uses `as unknown as LiveInstanceStatus` (the advertised `satisfies` check is fiction) over fixtures captured 2026-06-21 that are **seven required keys behind** the current Pydantic surface. Plus duplicate vocabularies across the API types (three process-state unions, three `sec_type` unions with divergent members, `PAPER_ONLY`/`paper-only` shipping side by side). **Fix: repoint the snapshot loop, add `tests/services` to CI, replace `as unknown as` with a real `satisfies`, re-capture fixtures.**

*Destructive-action UX is mostly right, with two gaps.* `mark_poisoned` uses a case-sensitive typed-`HALT` confirm re-guarded at open and confirm; `flatten_and_pause` re-checks eligibility at confirm time against the latest poll; all actions disable while any is busy (double-submit safe). **Gaps:** `stop` has *no* confirmation (one-click) despite being instance-retirement, and the **orders-page cancel is the weakest destructive path in the app** — one click, no confirm, no in-flight guard, client-inferred eligibility. Give both a backend `ActionCapability` + consequence-preview per §8.2.

*Timestamps are disciplined* (zero `new Date(string)` anywhere; all inputs ms-epoch) *except* unlabeled UTC ISO in audit panels and a date-only render on the 60 s-TTL account receipt's Generated/Expires — which hides the very time the staleness verdict depends on. Label it with the time.

## 9. Trader manual — proposed structure

The platform now has enough backend-authored vocabulary (`submit_readiness` codes, gate statuses, posture enums, notice tiers) that a manual can be written **in the system's own trader language** rather than inventing a parallel one. Structure below; each chapter names the backend contract it teaches so the manual can never drift silently from the product (same-PR rule as the authority doc).

**Part I — Mental model (concepts before screens)**
1. *What this platform is and is not.* Research/education; paper-first; live-money enablement explicitly out of scope today (`bot-lifecycle-account-owner-authority.md:22`). Nothing here is financial advice.
2. *The four independent facts about a bot.* PROCESS (is code running), INTENT (what you asked for: RUNNING/PAUSED/STOPPED), READINESS (can it act on the next bar), BROKER (what the broker session proves). Teaches: ADR 0013's Playwright meta-rule — there is deliberately **no single master status**, and why that is honest rather than confusing.
3. *What a bot is.* `strategy_instance_id` = durable name; a bot is a sealed instruction package once deployed (config changes = redeploy, ADR 0017). Names are never reused (ADR 0016 §11).
4. *What a run is.* Content-addressed `run_id` (ADR 0006); why an identical redeploy is the same run and any change is a new one; runs are evidence, instances are the spine (ADR 0004).
5. *What account ownership means.* One broker account, many bots: `bot_order_namespace = learn-ai/{instance}/v1` stamped on every order (`orderRef`); the account registry (`instance_registry.jsonl`); orphan / foreign / retired-owner-live-exposure classes in plain words (authority doc §4, :172).
6. *Safe to monitor vs safe to submit.* Direct mapping of `submit_readiness.code` values (authority doc :417): `safe_to_submit` (all ten conditions), `safe_to_monitor`, `blocked_before_submit`, `broker_state_unproven`, `account_frozen`, `waiting_for_owner_generation`, `submit_outcome_uncertain`. The manual's central table.
7. *Paper vs live vs observe.* `effective_posture` (PAPER_EXECUTION / PAPER_OBSERVATION / UNSAFE / UNKNOWN); why "Paper" radio historically meant *observe-only* (`readonly=true`) and how the Execution-capability labels (READ-ONLY OBSERVATION / PAPER ORDERS ENABLED) resolve it (attention PRD :36-37, :202-224).

**Part II — Operating procedures (task-shaped, one page each)**
8. *Create a validated bot package.* Spec + QC audit copy + backtest id + fixtures; what validation proves and does not prove ("operator-recorded", never "QC-approved" — ADR 0011 forbidden-labels list).
9. *Deploy.* Preconditions table from ADR 0006 (dirty tree, daemon down, collision) in trader words; account identity is derived from the connected session, never typed; sizing default = safe canary `FixedShares(1)` (ADR 0009) and what that means for the first session.
10. *Read Bot Control.* Pills, lifecycle chart, node inspector (receipts = evidence, actionability banner), Act-now bar. What each chart node means (the 9 canonical nodes, authority doc §3.1) — with the honest-gap list (which nodes can be `unknown` and why that is the system telling the truth).
11. *Read Account Monitor.* Account-level receipt (fresh/stale/TTL), freeze banner, connected-account proof; when account state overrides every bot's local view.
12. *Read Activity and Orders.* Broker rows vs engine expectations; verdict chips (`expected` / `expected_with_caveat` / `unexpected` / `engine_only_pending`); folds/clusters; why a fill can appear before the bot's own record (capture vs occurrence).
13. *Respond to warnings.* Notice tiers (info/warning/critical) and the contract that every critical carries an action; attention groups; "not proven" ≠ "broken" — what evidence is missing and how to supply it.
14. *Pause / Resume / Flatten-and-pause / Stop / Mark poisoned.* Exact semantics per ADR 0010 (Stop does not flatten; Flatten-and-pause = durable pause then one-shot flatten; Resume is guarded — the four gates); consequence preview before each.
15. *Recover.* Crash (watchdog sequence — what the system already did for you: submissions blocked, PAUSED persisted, flatten attempted, incident written — authority doc §7); reconnect (generation/drain phases); dirty account (freeze → recovery proof or audited override, §6.2); uncertain submit (why the system halted; probe outcomes PRESENT / PROVABLY_ABSENT / NOT_PROVABLE in trader words).
16. *When not to trade.* Restart-intensity freeze (3 starts / 5 min); session boundaries; degraded market data; `broker_state_unproven`; the rule that absence of proof blocks, and overrides are audited and expire.

**Part III — Reference**
17. *Glossary: backend concept → trader phrase.* Seeded from `operator-reason-codes.snapshot.json`, `GateResult.status` (pass/block/poison/freeze/unknown/not_applicable), lifecycle node ids, notice codes, submit_readiness codes, posture enums, reconciliation verdicts. Rule: the glossary is generated/checked against the closed backend vocabularies so it cannot drift (CI-diffable like the operator-reason-codes snapshot).
18. *Artifact atlas for the curious engineer-trader.* One page: which file answers which question, mirroring §3 of this report.
19. *Incident cookbook.* One entry per edge-case family in §7 with: what you see, what it means, what to click, what to never do.

**Copy principles** (details in §8): verbs for actions, states for facts; every warning names its evidence; every disabled control names its reason (backend-authored); never render a raw code as primary copy (receipts only); "not proven" language for absence-of-evidence; consequence preview on every destructive action; paper/observe/live mode always visible, never inferable.

**Frontend (Vitest — `podman exec my-frontend npx ng test --watch=false`):**
- Contract-spec diff: every backend closed vocabulary (`operator-reason-codes.snapshot.json`, `GateResult.status`, submit_readiness codes) has a frontend map entry; an unknown code renders through `receiptLabel`, never raw or silent (P2/row 28).
- No client-side operational judgment: assert the flagged `lib/` derivations (`account-summary-attention`, execution-uncertainty) are removed or backend-sourced (ADR 0013 debt); Playwright meta-rule holds (independent PROCESS/INTENT/READINESS/BROKER/SAFETY assertions, no synthetic master status).
- Destructive-action UX: Flatten/Stop consequence-preview modal renders; Mark-poisoned requires type-to-confirm; STOP banner resolves to a terminal state and degrades honestly on kill-during-stop (P1-6/row 23).
- Posture strip is always present and unmistakable; disabled controls show a backend-authored reason.

**Integration / replay / recovery (the highest-value, least-covered surface — prior audits show recovery paths are the most-broken):**
- Kill daemon with an open position + working order → assert the ordered halt artifacts (submits blocked → PAUSED → flatten proof *before* disconnect → incident) (row 1).
- Disconnect during `PENDING_INTENT` → reconnect → single classification, no double-submit (row 3); reconnect account mismatch → halt (row 8).
- Stale-snapshot flatten regression (fails before VCR-0019 fix) + partial-fill-during-flatten (row 5/P0-4).
- Restart data plane with a live child → live-binding recovery reads the durable registry, not a guess (row 31 / PR #764).
- Two simultaneous flatten requests → one effect (concurrent-operators, row 32) — will fail until `mutation_attempt_id` is enforced (documents the P1/ADR-0022 gap).

**Manual validation (operational confirmation, not CI):**
- Normal-market paper run: after a real broker fill, the fill appears in Bot Control Activity without manual reconstruction (ADR 0016 §"next live validation").
- **orderRef cap-C receipt:** place one live paper order, read back the stored `orderRef`, confirm IBKR echoes the full string untruncated — this is the blocking gate that keeps durable-submit activation refused (`order_identity.py:43-49`; ADR 0008 Acceptance Gate #1). IBKR documents no cap (§research), so this receipt is the only way to set C.
- Security: from a second LAN host, confirm the data-plane mutation endpoints are unreachable after P0-1; from a scratch webpage, confirm a `text/plain` POST is rejected after P0-2.

**Security note on the test environment:** several of these tests exercise broker-control endpoints; run them with `IBKR_READONLY=true` (the default kill switch) so a test can never place a real order, and against the FakeBroker where possible — but note that FakeBroker's `requires_durable_submit=False` historically masked VCR-0020, so the emergency-flatten smoke test must run against a broker double that enforces the `order_ref` invariant.

## 12. Proposed ADR amendments and new ADRs

**Amendments to existing ADRs** (exact titles):

1. **ADR 0001 amendment — "Projection trigger fired: Postgres lifecycle read model as a rebuildable projection".** Record: trigger 2 (cross-run operator queries became a hot path — timelines/safety-triage) fired 2026-06-30; shipped shape (5 tables, replay seam, file-locked cursor tailer, `LIFECYCLE_PROJECTION_ENABLED` fallback); the standing guarantees (files canonical, Postgres truncatable, `/status` never depends on it); and the rule for future projection slices (each names its trigger + fallback in this ADR's ledger). 7. **ADR 0010/0017 joint amendment — "Canonical render-site registry for destructive actions".** One table: action → surface(s) → placement → confirmation style; resolves Stop/mark-poisoned placement drift and the dashboard PRD deltas.

9. **ADR 0015 corrigendum** — fix `"focus_bot control_action"` literal to match code; reconcile Supersedes/extends header.
10. **ADR 0016 amendment — "Package registry status"** — either schedule the immutable package registry slice or narrow the ADR's fail-closed claim to what shipped (drift-rejection on loose fields).

**New ADRs to write** (exact titles + reason):

- **ADR 0018 — "Account-scoped ownership and fail-closed account reconciliation receipts".** Consolidates what already shipped across authority doc §4/§6 and the triage PRD (instance_registry.jsonl as the durable namespace source; Account Truth attribution from the registry incl. `retired_owner_live_exposure`; account-level receipt with TTL + connected-account proof + `gate_id=account.reconciliation`; freeze/clear via recovery proof XOR audited override). - **ADR 0022 — "Daemon-enforced mutation idempotency"** (0010:242 debt): `mutation_attempt_id` graduates from audit-only to enforced dedup key at the daemon.
