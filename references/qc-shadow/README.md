# QC Shadow — QuantConnect Cloud parallel observer

This directory carries the **audit copy** of the QC algorithm and the
**checked-in QC Cloud exports** that the IBKR paper-shadow deployment
reconciles against per spec
[`docs/superpowers/specs/2026-05-08-ibkr-paper-shadow-deployment-design.md`](../../docs/superpowers/specs/2026-05-08-ibkr-paper-shadow-deployment-design.md).

## What lives here

```
references/qc-shadow/
├── README.md                              ← this file
├── SpyEmaCrossoverAlgorithm.py            ← audit copy (§ 3 source of truth)
└── backtests/
    ├── lean-parity-fixture/               ← Test 1 export (§ 8.1)
    │   ├── trades.csv
    │   ├── indicators.csv
    │   └── attribution.md                 ← QC backtest id, run date, fixture sha
    └── 2025-08-01_to_2025-11-01/          ← Test 2 export (§ 8.2)
        ├── trades.csv
        ├── indicators.csv
        ├── bars.csv
        └── attribution.md
```

`backtests/` subdirectories arrive as one-time exports from QC Cloud
when the operator runs each test there. They are **not** generated by
this repo's CI; the operator-facing workflow below produces them.

## Audit vs execution copy

`SpyEmaCrossoverAlgorithm.py` here is the *audit copy*. The QC Cloud
workspace runs an identical *execution copy* of this file. Before each
run, the operator records the SHA-256 of this audit copy and the QC
Cloud backtest id in the run ledger (§ 10) — mismatch halts the
pre-paper gate.

`from AlgorithmImports import *` only resolves inside QC Cloud, so this
file does **not** import in this repo's pytest. CI does not execute it.

## Operator workflow — Test 1 (same-bar engine parity, § 8.1)

This is the **strict** parity test. Both engines consume the same
LEAN-formatted minute-bar fixture; indicators must match to `atol=1e-9`
and trades must match exactly.

The QC custom-data harness is the critical-path piece (§ 8.1). Without
it there is no clean engine-equivalence proof.

1. Build a custom-data replay class in QC Cloud that:
   * Loads the SPY minute fixture used by `tests/test_spy_validation.py`
     (LEAN format, `2024-03-28 → 2026-03-27`).
   * Disables QC's native data feed for the duration of the backtest.
   * Verifies the QC algorithm receives exactly the bar count and
     timestamps the Python fixture provides.
2. Upload `SpyEmaCrossoverAlgorithm.py` from this directory verbatim.
3. Run the QC backtest against the custom-data feed.
4. Export to `references/qc-shadow/backtests/lean-parity-fixture/`:
   * `trades.csv` — one row per QC trade with columns
     `entry_time_ms,exit_time_ms,entry_price,exit_price,pnl_points`.
   * `indicators.csv` — one row per consolidated 15-min bar with
     columns `bar_close_ms,ema5,ema10,rsi,signal`
     (`signal ∈ {ENTER, EXIT, HOLD}`).
   * `attribution.md` — record QC Cloud backtest id, run date, the
     SHA-256 of `SpyEmaCrossoverAlgorithm.py` at upload time, and the
     SHA-256 of the fixture file the harness loaded.
5. Commit the export.
6. `python -m pytest tests/engine/live/test_qc_python_parity_fixture.py`
   should now pass.

If the harness cannot be built, see § 8.1's "Fallback decision point"
— do not auto-substitute native QC data.

## Operator workflow — Test 2 (native-feed shadowing, § 8.2)

This is the **operational** test. QC runs against its native data feed,
Python runs against Polygon historical, and the comparison uses
cross-feed tolerances (EMA atol $0.10, RSI atol 2.0).

1. Upload the same `SpyEmaCrossoverAlgorithm.py` to QC Cloud.
2. Configure the QC backtest:
   * Resolution: minute SPY (the algorithm consolidates internally).
   * Window: `2025-07-15` (warmup pre-roll start) → `2025-11-01` half-open
     (evaluation end). RTH only; no extended hours.
3. Run the backtest in QC Cloud.
4. Export to `references/qc-shadow/backtests/2025-08-01_to_2025-11-01/`:
   * `trades.csv` (same schema as Test 1).
   * `indicators.csv` (same schema as Test 1).
   * `bars.csv` — one row per 15-min consolidated bar from QC's data
     feed: `bar_close_ms,open,high,low,close,volume`.
   * `attribution.md` — same fields as Test 1, plus the QC data-feed
     identifier (e.g. `Algoseek US Equities`).
5. Commit the export.
6. `python -m pytest tests/engine/live/test_qc_python_native_feed.py`
   should pass with `0` engine-class divergences. Data-class
   divergences are reported and require operator sign-off.

## Operator workflow — Paper week (§ E)

During the 15 RTH paper days, QC Cloud is run in paper-trading mode
against its own data feed.

* **No IBKR connection from QC.** The QC paper sim must not be
  configured to deploy to IBKR — see § 5 isolation invariant.
* End-of-day, export QC's per-day trade log + per-bar indicators and
  drop them under `PythonDataService/artifacts/qc/<YYYY-MM-DD>/`:
  * `trades.csv`
  * `indicators.csv`
* The next morning's reconciliation (`python -m app.engine.live.reconcile`)
  hashes both files into the day-N Markdown manifest and verifies
  them on the following morning's pre-flight (§ 6.4 #7,
  see `app.engine.live.pre_flight.check_yesterday_artifacts_valid`).
  A missing or mismatched export halts that morning's session.

## Schema details — trades.csv

| Column            | Type   | Notes |
|---|---|---|
| `entry_time_ms`   | int64  | Bar that triggered entry; canonical ms UTC |
| `exit_time_ms`    | int64  | Bar that triggered exit |
| `entry_price`     | float  | QC fill price |
| `exit_price`      | float  | QC fill price |
| `pnl_points`      | float  | `exit_price − entry_price`; QC's bookkeeping |

## Schema details — indicators.csv

| Column            | Type   | Notes |
|---|---|---|
| `bar_close_ms`    | int64  | Consolidated 15-min bar end, canonical ms UTC |
| `ema5`            | float  | EMA(5), updated at bar close |
| `ema10`           | float  | EMA(10), updated at bar close |
| `rsi`             | float  | RSI(14, Wilders) |
| `signal`          | str    | One of `ENTER` / `EXIT` / `HOLD` |

A row appears for every consolidated bar — including warmup bars (the
test consumers handle the warmup gating).

## Why no QC API automation in v1

§ 13 #7 of the spec accepts manual QC Cloud UI export for v1; QC API
automation (using `lean.io`'s REST surface) is a Phase B sub-task to
revisit if 15 days of manual export becomes painful.
