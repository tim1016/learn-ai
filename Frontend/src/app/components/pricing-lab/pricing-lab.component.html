Pricing Lab — Documentation

What Is the Pricing Lab?

The Pricing Lab is a validation study for option pricing. It computes the theoretical price and Greeks (Delta, Gamma, Theta, Vega, Rho) for a single option contract using 8 independent pricing models across a range of underlying spot prices. By overlaying all curves on one chart, you can visually verify that the models agree — and investigate where and why they diverge.

Testing Steps

Step 1 — Load a ticker: Enter the underlying symbol (e.g., SPY, AAPL, QQQ) and click Load. This fetches all available option expirations from the market data API.

Step 2 — Pick a contract: Choose an expiration date, select Calls or Puts, then pick a strike from the dropdown. The dropdown shows strike price and implied volatility (IV) for each contract. Contracts with higher IV will produce larger price values. The comparison begins automatically when you select a contract.

Step 3 — Read the comparison chart: The top chart overlays all 8 models. If lines overlap perfectly, the models agree. Use the Metric toggle to switch between Price, Delta, Gamma, Theta, Vega, and Rho. Hover over the chart to see exact values for each model at any spot price.

Step 4 — Read the difference chart: The bottom chart shows the numerical difference between a reference model and every other model. Choose which model is the reference via the Diff Reference dropdown. When all models agree closely (differences < 0.001), the y-axis auto-scales with a multiplier (e.g., ×10,000) so tiny differences become visible.

Step 5 — Adjust parameters: Change the Risk-Free Rate (default 5%) or Range % (how far above/below the current spot price to plot, default ±20%). Click Compare to re-run with updated parameters.

Step 6 — Toggle models: Click the model chips (LBS, PyBS, QL-A, CRR, JR, LR, FD, MC) to show/hide individual models. Crossed-out labels = hidden. This helps isolate specific model pairs.

What Was Implemented

The Pricing Lab evaluates a single European option at 100 evenly-spaced spot prices (from spot × (1 − range%) to spot × (1 + range%)). For each spot price, every pricing model computes: the option price (NPV), and all five first-order Greeks.

The Legacy BS model runs entirely in the browser (client-side JavaScript). All other models are computed on the server via a single GraphQL query (pricingModelComparison) that calls the Python data service endpoint POST /api/quantlib/compare. This endpoint returns curves for Python BS and all 6 QuantLib engines in one response.

If any individual engine fails for a given contract (e.g., numerical instability), that engine is skipped gracefully while others continue — the comparison is never all-or-nothing.

Mathematical Calculations

Black-Scholes closed-form: For a European call, C = S·N(d1) − K·e−rT·N(d2), where d1 = [ln(S/K) + (r + σ²/2)T] / (σ√T) and d2 = d1 − σ√T. The three analytic models (Legacy BS, Python BS, QuantLib Analytic BS) all use this formula — they differ only in how N(x) (the standard normal CDF) is computed.

Legacy BS normCdf: Uses the Abramowitz & Stegun (1964) polynomial approximation (formula 7.1.26) with 5 coefficients. Maximum absolute error |ε| < 1.5×10−7. This is the same implementation used in the live option chain pricing.

Python BS normCdf: Uses scipy.stats.norm.cdf which provides full IEEE 754 double-precision accuracy (~15 significant digits).

QuantLib Analytic BS: Uses AnalyticEuropeanEngine with a GeneralizedBlackScholesProcess. The CDF is computed via the C++ error function implementation, also at full double precision.

Binomial trees (CRR, JR, LR): Discretize the stock price evolution into a recombining tree. CRR uses u = eσ√Δt, d = 1/u. JR uses drift-adjusted u = e(r − σ²/2)Δt + σ√Δt. LR uses the Peizer-Pratt inversion for transition probabilities, achieving faster convergence. The comparison uses 201 time steps (the single-option endpoint uses 801).

Finite Differences: Solves the Black-Scholes PDE numerically on a 201×200 space×time grid using the Crank-Nicolson scheme (second-order accurate in both space and time). The single-option endpoint uses an 801×800 grid for higher accuracy.

Monte Carlo: Simulates pseudorandom price paths under the risk-neutral measure (geometric Brownian motion). Uses seed=42 for reproducibility. The comparison uses tolerance=0.02 (the single-option endpoint uses 0.001). Stochastic noise in the MC curve is expected and intentional — it demonstrates the probabilistic nature of the method.

Greeks Calculations & Conventions

Delta = ∂V/∂S — sensitivity to underlying price. Range: [0, 1] for calls, [−1, 0] for puts. Computed analytically where possible.

Gamma = ∂²V/∂S² — rate of change of delta. Always positive for vanilla options. Highest near ATM and expiration.

Theta = ∂V/∂t — time decay. Reported per calendar day (annual value ÷ 365). Usually negative for long options (the option loses value as time passes).

Vega = ∂V/∂σ — sensitivity to implied volatility. Reported per 1% IV move (annual value ÷ 100). Highest for ATM, long-dated options.

Rho = ∂V/∂r — sensitivity to the risk-free rate. Reported per 1% rate move (annual value ÷ 100).

For engines without analytic Greek support (Monte Carlo, Finite Differences), Greeks are computed via numeric finite differencing (bump-and-revalue): Δ ≈ [V(S+h) − V(S−h)] / 2h. In the comparison endpoint, theta, vega, and rho may fall back to 0 for slow engines to keep computation time reasonable.

How to Interpret Results

Analytic models agree? The three BS variants (Legacy, Python, QuantLib) should produce near-identical curves. Differences arise purely from normal CDF precision. If max difference is < 1×10−5, the implementations are validated.

Binomial models converge? CRR, JR, and LR should be very close to the analytic price. Small oscillations (especially CRR) are expected due to tree discretization. LR typically converges fastest. If you see large differences, the option may have extreme parameters (very short expiry, very deep ITM/OTM).

FD matches analytic? Finite differences should closely match BS for European options. Its real value is for American options or exotic payoffs where closed-form solutions don't exist.

MC has noise? Expected. Monte Carlo produces a noisy curve because each spot point is a separate simulation. The noise band width depends on the tolerance parameter. With tol=0.02 (comparison mode), expect ±0.02 noise around the true price.

Red flags: If any model diverges significantly (e.g., >$0.10 for price or >0.01 for delta), investigate the contract parameters. Extreme IV (>200%), very short expiry (<1 day), or deep ITM/OTM contracts can stress numerical methods.

System Architecture

Frontend (Angular): Computes Legacy BS client-side using black-scholes.ts (Abramowitz & Stegun normCdf polynomial). Sends a single GraphQL query for all server-side models. Charts rendered with lightweight-charts v5 (TradingView).

Backend (.NET / Hot Chocolate): GraphQL resolver pricingModelComparison proxies the request to the Python data service via POST /api/quantlib/compare. Uses JsonNamingPolicy.SnakeCaseLower for Python interop. Timeout: 300 seconds.

Data Service (Python / FastAPI): Endpoint /api/quantlib/compare iterates all engines for each spot price. Python BS uses scipy.stats.norm. QuantLib engines use compiled C++ via SWIG Python bindings (QuantLib-Python). Graceful degradation: if one engine fails, others continue.
